### Architecture Note: The Solana Verification Bridge

**This is the most critical architectural decision in the entire project.**

EZKL generates Halo2/KZG proofs and only natively supports EVM verification (Solidity verifier contracts). Solana does NOT have a native Halo2 verifier. Solana supports Groth16 (BN254) verification via alt_bn128 precompiles.

**The recommended pipeline for dual-chain verification:**

1. **Train model** (PyTorch) --> **Export** (ONNX) --> **Generate circuit + proof** (EZKL/Halo2)
2. **Ethereum path:** EZKL `create-evm-verifier` --> deploy Solidity verifier --> verify on-chain (native support)
3. **Solana path:** Wrap EZKL proof into Groth16 via SP1 or RISC Zero Bonsai --> verify via groth16-solana on-chain

**Alternative Solana path (simpler but less elegant):** Verify proof off-chain, publish only the result + proof hash on Solana, with Ethereum as the canonical verification chain. Solana stores the score + a pointer to the Ethereum verification transaction.

**Confidence:** LOW on the SP1 wrapping path (novel integration, limited examples of EZKL+SP1). MEDIUM on the hybrid verification path.

## Alternatives Considered

| Recommended | Alternative | When to Use Alternative |
|-------------|-------------|-------------------------|
| EZKL (Halo2) | Risc Zero zkVM | If you need to prove arbitrary Rust code execution (not just ML inference). Risc Zero is a general-purpose zkVM, not ML-specialized. 65x slower for ML, but more flexible for non-ML logic. Consider if evaluation logic is complex beyond ML. |
| EZKL (Halo2) | Lagrange DeepProve | If you need to prove larger models (GPT-2 scale). Claims 54-158x faster than EZKL. BUT: not privacy-preserving, runs on EigenLayer (adds dependency), less mature ecosystem. |
| EZKL (Halo2) | Jolt Atlas (ICME/NovaNet) | If EZKL hits model-size limits. 4-7x faster than prior SOTA, supports true zero-knowledge. BUT: newer, less battle-tested, smaller community. |
| Foundry | Hardhat | If the team is JS-heavy and needs extensive plugin ecosystem. But Foundry is 5-10x faster and the industry is converging on it. |
| LiteSVM | solana-test-validator | If you need full Solana runtime fidelity (RPC, block production). But LiteSVM is orders of magnitude faster for unit/integration tests. Use solana-test-validator only for final E2E tests. |
| Wormhole | LayerZero | If you need omnichain messaging beyond Solana+Ethereum. LayerZero covers 50+ chains vs Wormhole's 35+. But Wormhole has deeper Solana integration and is the standard for Solana cross-chain. |
| SP1 (Succinct) | RISC Zero Bonsai | For STARK-to-Groth16 wrapping. Bonsai is a hosted prover service that can produce Groth16 SNARKs. Better if you want managed infrastructure. BUT: adds external dependency (hosted service), costs money per proof. |

## Stack Patterns by Variant

**If zkML proof generation is too slow for real-time evaluation:**
- Use async evaluation pattern: proposal submitted on-chain --> event triggers off-chain EZKL proof generation (may take 30s-5min) --> proof + score submitted on-chain in a separate transaction
- This is likely the production pattern regardless. Real-time zkML (<100ms) is not achievable in 2025/2026 for meaningful models.

**If EZKL cannot handle the evaluation model's complexity:**
- Split into micro-models: one for security analysis, one for fraud detection, one for impact scoring, one for alignment scoring
- Each micro-model produces its own proof
- Composite scoring contract aggregates verified sub-scores on-chain
- This also improves transparency (users see per-criterion proofs)

**If Solana Groth16 wrapping proves impractical:**
- Use Ethereum as the canonical verification chain
- Solana program stores: score, reasoning hash, Ethereum tx hash pointing to verification
- Cross-chain score attestation via Wormhole message
- This is architecturally simpler and may be the pragmatic Phase 1 approach

**If multi-modal input (code analysis + text + demo) is too complex for a single zkML model:**
- Pre-process inputs off-chain into structured feature vectors
- Prove the scoring model (feature vectors --> scores) via zkML
- The pre-processing step is NOT proven (acknowledged trust assumption)
- Document this tradeoff clearly -- partial verifiability is better than no verifiability

## Critical Constraints

### zkML Model Design Constraints

1. **ONNX operator coverage**: EZKL supports ~50 of 120+ ONNX operators. Design models using supported ops only. Test ONNX export BEFORE building the full pipeline.
2. **Quantization**: All floating-point operations are converted to fixed-point arithmetic in the zk circuit. Expect slight accuracy drift. Use larger bit widths (slower proofs) or accept minor scoring variance.
3. **Activation functions**: ReLU is cheap. Sigmoid/tanh require exponentiation (expensive). GELU/softmax (common in transformers) are very costly or unsupported. Design models with ReLU or piecewise-linear activations.
4. **Model size**: Current practical limit is small-to-medium models. GPT-2 scale is at the frontier. Design evaluation models to be small and specialized.
5. **Non-linear operations**: Avoid where possible. Every non-linearity adds circuit complexity and proof time.

### Multi-Chain Deployment Constraints

1. **Data consistency**: Proposal data must be canonical on one chain and referenced on the other. Recommend Ethereum as canonical (cheaper storage for proof verification data), Solana for fast proposal submission.
2. **Cross-chain latency**: Wormhole messages take 15-20 minutes for finality. Score publication will not be instant across chains.
3. **Cost**: Ethereum L1 verification is expensive (~270k gas for Groth16 verify). Consider L2 (Base, Arbitrum) for the Ethereum verifier.
