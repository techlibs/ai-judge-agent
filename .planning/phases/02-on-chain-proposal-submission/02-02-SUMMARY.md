# Phase 02 Plan 02: Solana Anchor Proposal Submission Summary

Anchor program with submit_proposal, initialize_config, and update_status instructions using PDA-based proposal accounts with Config PDA enforcing evaluator authority per D-10.

## What Was Built

### Anchor Program (programs/grant-evaluator/src/lib.rs)

Three instructions implemented:

1. **initialize_config** -- Creates a Config PDA (seeded `[b"config"]`) storing the authorized evaluator pubkey and deployment authority. Called once after deployment.

2. **submit_proposal** -- Any wallet can submit a proposal. Creates a Proposal PDA seeded with `[b"proposal", submitter, title_hash]`. Stores content_cid, repo_url_hash, demo_url_hash, domain, status (Submitted), and submitted_at timestamp. Validates content_cid is non-zero. Emits `ProposalSubmitted` event.

3. **update_status** -- Only the evaluator address stored in Config PDA can transition proposal status. Enforces linear status progression: Submitted -> UnderReview -> Evaluated -> Disputed. Emits `ProposalStatusChanged` event.

### Data Model

- **Config**: authority (Pubkey), evaluator (Pubkey), bump (u8) -- 65 bytes + 8 discriminator
- **Proposal**: submitter, content_cid, repo_url_hash, demo_url_hash, domain, status, submitted_at, bump -- 139 bytes + 8 discriminator
- **ProposalDomain**: DeFi, Governance, Education, Health, Infrastructure, Other
- **ProposalStatus**: Submitted, UnderReview, Evaluated, Disputed

### Integration Tests (tests/anchor/grant-evaluator.test.ts)

10 tests covering all behaviors:
1. PDA creation with correct fields
2. ProposalSubmitted event emission
3. Zero content_cid rejection
4. Duplicate proposal rejection (PDA collision)
5. Open submission (multiple wallets)
6. Config initialization with evaluator authority
7. Authorized status transition (Submitted -> UnderReview)
8. Unauthorized signer rejection (D-10 enforcement)
9. Invalid status transition rejection
10. All 6 domain values accepted
