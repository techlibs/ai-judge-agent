**Create `src/lib/chain/reputation-schemas.ts`:**

Define Zod schemas for reputation data at the API boundary, per CLAUDE.md "Zod validation at boundaries" rule. These schemas validate the data returned from viem readContract calls before it leaves the API route.

```typescript
import { z } from "zod";

export const reputationFeedbackEntrySchema = z.object({
  clientAddress: z.string(),
  value: z.number(),
  tag1: z.string(),
  tag2: z.string(),
  feedbackURI: z.string(),
  feedbackHash: z.string(),
  blockNumber: z.number().int().nonnegative(),
  timestamp: z.number().int().nonnegative(),
  txHash: z.string().regex(/^0x[a-fA-F0-9]{64}$/).nullable(),
});

export const reputationSummarySchema = z.object({
  feedbackCount: z.number().int().nonnegative(),
  averageScore: z.number().min(0).max(100),
});

export const reputationResponseSchema = z.object({
  tokenId: z.string(),
  summary: reputationSummarySchema,
  history: z.array(reputationFeedbackEntrySchema),
});

export type ReputationFeedbackEntry = z.infer<typeof reputationFeedbackEntrySchema>;
export type ReputationSummary = z.infer<typeof reputationSummarySchema>;
export type ReputationResponse = z.infer<typeof reputationResponseSchema>;
```

**Create `src/lib/chain/reputation.ts`:**

This file provides three functions that read reputation data from the on-chain ReputationRegistry using viem.

Import the existing `publicClient` from `src/lib/chain/client.ts` and `REPUTATION_REGISTRY_ADDRESS` from `src/lib/chain/contracts.ts`. Import the ReputationRegistry ABI (same ABI already used in Phase 2 storage.ts for `giveFeedback` writes). Import types from `reputation-schemas.ts`.

**`getReputationHistory(agentId: string): Promise<Omit<ReputationFeedbackEntry, "txHash">[]>`**

1. Call `publicClient.readContract({ address: REPUTATION_REGISTRY_ADDRESS, abi: reputationRegistryAbi, functionName: "readAllFeedback", args: [BigInt(agentId)] })`.
2. The return is an array of feedback structs. Map each to a partial entry (without txHash -- that comes from getTxHashesForBlocks):
   - `value`: Convert from fixed-point. The contract stores `int128 value` with `uint8 valueDecimals`. Compute: `Number(feedback.value) / Math.pow(10, Number(feedback.valueDecimals))`.
   - `blockNumber`: `Number(feedback.blockNumber)`.
   - `feedbackHash`: Convert `bytes32` to hex string.
   - `feedbackURI`, `tag1`, `tag2`, `clientAddress`: direct string values.
3. For each entry, fetch the block timestamp: `const block = await publicClient.getBlock({ blockNumber: BigInt(entry.blockNumber) }); entry.timestamp = Number(block.timestamp);`
   - Optimization: batch unique block numbers to avoid duplicate getBlock calls. Create a `Map<number, number>` of blockNumber->timestamp, fetch each unique block once.
4. Sort entries by blockNumber descending (newest first).
5. Return the array.
6. Wrap the entire function in try/catch. On error, log the error and return an empty array (graceful degradation -- the UI shows an empty state).

**`getReputationSummary(agentId: string): Promise<ReputationSummary>`**

1. Call `publicClient.readContract({ address: REPUTATION_REGISTRY_ADDRESS, abi: reputationRegistryAbi, functionName: "getSummary", args: [BigInt(agentId), [], "grant-evaluation", ""] })`.
   - First arg: agentId as BigInt.
   - Second arg: empty array for clientAddresses (query all).
   - Third arg: `"grant-evaluation"` (tag1 used in Phase 2 publishScoreOnChain).
   - Fourth arg: `""` (tag2 empty for any version).
2. The return is `[count: bigint, summaryValue: bigint, summaryValueDecimals: number]`.
3. Convert: `feedbackCount = Number(count)`, `averageScore = Number(summaryValue) / Math.pow(10, Number(summaryValueDecimals))`.
4. Return `{ feedbackCount, averageScore }`.
5. On error, return `{ feedbackCount: 0, averageScore: 0 }`.

**`getTxHashesForBlocks(agentId: string, blockNumbers: number[]): Promise<Map<number, string>>`**

This function recovers the actual transaction hashes from on-chain event logs. Phase 2's `publishScoreOnChain` calls `giveFeedback` which emits a `FeedbackGiven` event. We query these events by block number to get the real txHash.

1. Create a `Map<number, string>` to store blockNumber -> txHash mappings.
2. Deduplicate blockNumbers: `const uniqueBlocks = [...new Set(blockNumbers)]`.
3. Import `parseAbiItem` from viem. Define the event signature: `const feedbackGivenEvent = parseAbiItem("event FeedbackGiven(uint256 indexed tokenId, address indexed evaluator, uint256 score, string contentHash)")`.
4. For each unique block number, call `publicClient.getLogs({ address: REPUTATION_REGISTRY_ADDRESS, event: feedbackGivenEvent, args: { tokenId: BigInt(agentId) }, fromBlock: BigInt(blockNum), toBlock: BigInt(blockNum) })`.
5. If logs are returned, extract `logs[0].transactionHash` and store in the map.
6. Wrap in try/catch per block -- if a single block's log query fails, skip it (txHash will be null in the response).
7. Return the map.

**Create `src/app/api/reputation/[tokenId]/route.ts`:**

```typescript
import { type NextRequest, NextResponse } from "next/server";
import { getReputationHistory, getReputationSummary, getTxHashesForBlocks } from "@/lib/chain/reputation";
import { reputationResponseSchema } from "@/lib/chain/reputation-schemas";
```

**`export async function GET(request: NextRequest, { params }: { params: Promise<{ tokenId: string }> }): Promise<NextResponse>`**

1. Extract `tokenId` from `await params`. Validate it is a non-empty string that can be parsed as a positive integer: `const id = (await params).tokenId; if (!/^\d+$/.test(id)) return NextResponse.json({ error: "Invalid tokenId" }, { status: 400 });`
2. Call all three functions: first get history and summary in parallel: `const [history, summary] = await Promise.all([getReputationHistory(id), getReputationSummary(id)]);`
3. Then recover txHashes for all blocks in the history: `const blockNumbers = history.map(e => e.blockNumber); const txHashMap = await getTxHashesForBlocks(id, blockNumbers);`
4. Enrich history entries with txHashes: `const enrichedHistory = history.map(entry => ({ ...entry, txHash: txHashMap.get(entry.blockNumber) ?? null }));`
5. Build response object: `const responseData = { tokenId: id, summary, history: enrichedHistory };`
6. Validate through Zod before returning: `const validated = reputationResponseSchema.parse(responseData);`
7. Create response: `const response = NextResponse.json(validated);`
8. Add Cache-Control header: `response.headers.set("Cache-Control", "public, s-maxage=30, stale-while-revalidate=60");` (30s fresh, 60s stale-while-revalidate -- satisfies CHAIN-05 "indexed or read cache" requirement by caching at the CDN/edge layer).
9. Return response.
10. On error: `NextResponse.json({ error: "Failed to fetch reputation data" }, { status: 500 })`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> API route | User-supplied tokenId in URL path parameter |
| API route -> Base Sepolia RPC | Server reads from public blockchain via RPC |
| API route -> Response | On-chain data validated through Zod before returning |
| UI -> Block explorer | External link to sepolia.basescan.org using real txHash |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | API route tokenId param | mitigate | Validate tokenId is numeric-only via regex `/^\d+$/` before passing to viem BigInt conversion; reject non-numeric with 400 |
| T-03-02 | Information Disclosure | RPC URL in client code | mitigate | Use NEXT_PUBLIC_RPC_URL env var; do not hardcode RPC endpoints or API keys in source |
| T-03-03 | Denial of Service | Excessive RPC calls | mitigate | Add Cache-Control header (s-maxage=30, stale-while-revalidate=60) on API response; page.tsx uses fetch with next.revalidate=30 for ISR caching |
| T-03-04 | Spoofing | Fake block explorer link | accept | We construct the URL deterministically from `sepolia.basescan.org/tx/{txHash}` where txHash is recovered from on-chain event logs -- no user-supplied URL injection. Low risk as basescan is the canonical explorer. |
| T-03-05 | Information Disclosure | Private key in reputation read | accept | Reputation reads use publicClient only (no wallet/private key needed). No secrets involved in read operations. |
| T-03-06 | Denial of Service | Large readAllFeedback result | accept | For v1, evaluation count per project will be small (< 100). Pagination via "Load more" button at 20 entries per UI-SPEC. Re-evaluate if production volume increases. |
| T-03-07 | Tampering | Unvalidated chain data | mitigate | All readContract return values parsed through Zod schemas (reputationResponseSchema.parse) before leaving the API route boundary |
</threat_model>
