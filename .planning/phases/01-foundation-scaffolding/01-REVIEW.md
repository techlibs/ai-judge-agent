# Phase 01: Code Review Report

**Reviewed:** 2026-04-14T03:17:02Z
**Depth:** standard
**Files Reviewed:** 10

## Summary

Reviewed the foundation scaffolding for the IPE City Grant Evaluator Agent — CI workflow, Anchor/Solana program scaffold, Solidity contract scaffold, build tooling (Makefile, foundry.toml), and package configuration.

The scaffold is minimal and correct for its purpose. No security vulnerabilities or data-loss risks were found. Two warnings require action before real code is layered on top: a significant TypeScript client version mismatch against the on-chain Anchor 1.0.0 program, and the Solana CI job never running tests. Four informational items are noted for tracking.

## Warnings

### WR-01: Anchor TypeScript client version mismatch (0.32 vs 1.0)

**File:** `package.json:14`
**Issue:** `@coral-xyz/anchor` is pinned to `^0.32.1` while `Anchor.toml` and `Cargo.toml` both declare Anchor `1.0.0` for the on-chain program. Anchor 1.0.0 introduced breaking changes to the IDL format and TypeScript client API. A 0.32.x client cannot correctly serialize/deserialize accounts or instructions generated by a 1.0.0 program. Any integration test or client code written against this dependency will silently fail or error at runtime when it calls the program.
**Fix:**
```json
"devDependencies": {
  "@coral-xyz/anchor": "^1.0.0",
  "@solana/web3.js": "^2.0.0",
  "typescript": "^5.7.0"
}
```
Note: `@solana/web3.js` v2.x is also recommended per the project stack (CLAUDE.md); upgrading both together avoids a second round-trip.

---

### WR-02: Solana program tests never run in CI

**File:** `.github/workflows/ci.yml:30-34`
**Issue:** The `solana` CI job builds the program and runs Clippy, but never executes `anchor test` or any equivalent test command. The Ethereum job correctly runs `forge test`, but Solana has no test execution. Any bugs introduced into the Anchor program will pass CI undetected until manual testing.
**Fix:** Add a test step after the build step in the `solana` job:
```yaml
- name: Run Anchor tests
  run: anchor test --skip-build
```
`--skip-build` avoids rebuilding what was already built in the prior step. When LiteSVM-based tests are added (per the project stack), they will run under `bun test` which is already the `[scripts] test` entry in `Anchor.toml` — so `anchor test` will invoke the correct runner.

## Info

### IN-01: `version` should be a constant in GrantEvaluator.sol

**File:** `contracts/src/GrantEvaluator.sol:7`
**Issue:** `uint256 public version` is set once in the constructor and never mutated. Declaring it as a storage variable wastes a 20,000-gas SSTORE on deployment and costs an extra SLOAD on every read. It should be an immutable constant.
**Fix:**
```solidity
uint256 public constant VERSION = 1;

// Remove constructor entirely, or if a constructor is needed for future setup:
constructor() {}
```

---

### IN-02: `@solana/web3.js` is on the deprecated v1.x series

**File:** `package.json:15`
**Issue:** `@solana/web3.js@^1.98.4` is the legacy v1 API. The project's own stack documentation (CLAUDE.md) recommends v2.x for new projects. This is not a breaking issue at scaffold stage but will require a migration before any client-side code is written against it, and that migration is non-trivial.
**Fix:** Upgrade to `@solana/web3.js@^2.0.0` alongside the Anchor client upgrade in WR-01.

---

### IN-03: `make clean` does not run `cargo clean`

**File:** `Makefile:27-30`
**Issue:** The `clean` target runs `anchor clean` and `forge clean` but not `cargo clean`. `anchor clean` removes the `.anchor/` directory and program keypairs but does not clean the Rust `target/` directory (which can reach several GB). Developers expecting a full clean will be surprised.
**Fix:**
```makefile
clean:
	anchor clean
	cd contracts && forge clean
	cargo clean --workspace
	rm -rf node_modules dist
```

---

### IN-04: Empty `Initialize` accounts struct lacks documentation of intent

**File:** `programs/grant-evaluator/src/lib.rs:15-16`
**Issue:** The `Initialize` struct is empty and the `initialize` instruction only logs a message. For a scaffold this is acceptable, but there is no comment indicating what accounts this instruction will eventually require. Future contributors will not know whether the emptiness is intentional (the instruction truly needs no accounts) or a placeholder.
**Fix:** Add a comment:
```rust
/// Accounts required for the initialize instruction.
/// TODO: Add evaluator config account, authority, and system program
/// when the EvaluatorConfig state is defined in Phase 2.
#[derive(Accounts)]
pub struct Initialize {}
```
